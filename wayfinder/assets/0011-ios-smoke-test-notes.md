## TL;DR — the iOS-safe export, knob by knob

Everything ticket 0002 §8 asked for, and exactly where it lives:

| 0002 §8 setting | File · key | Note |
| --- | --- | --- |
| Compatibility / WebGL2 | `project.godot` · `rendering/renderer/rendering_method="gl_compatibility"` | Forced on web regardless; set it anyway so the editor matches. |
| **Thread Support OFF** | `export_presets.cfg` · `variant/thread_support=false` | The one load-bearing toggle. Off ⇒ no `SharedArrayBuffer` ⇒ **no COOP/COEP headers** ⇒ runs on plain static hosts and dodges Safari's threaded-WASM bugs. |
| PWA ON | `export_presets.cfg` · `progressive_web_app/enabled=true` + `display=1` + `icon_144/180/512` | Godot emits `index.manifest.json` + `index.service.worker.js`. `display=1` = standalone (chrome-less WebClip on iOS). |
| Audio Sample | `project.godot` · `audio/general/default_playback_type.web=2` | Also forced per-node (`AudioStreamPlayer.playback_type = AudioServer.PLAYBACK_TYPE_SAMPLE`). |
| Basis Universal / mobile textures | `export_presets.cfg` · `vram_texture_compression/for_mobile=true` + `project.godot` · `textures/vram_compression/import_etc2_astc=true` | Both needed together, else the mobile-compressed variant isn't produced. |
| Modest WASM max memory | `export_presets.cfg` · `html/head_include` (a `<script>` shim) | 4.3 has **no preset knob** for this — see "The WASM-memory lever" below. |
| HTTPS, no COOP/COEP | GitHub Pages | Pages is HTTPS by default; we set no special headers. |

## 1. `export_presets.cfg` is just a text file (and it's the source of truth for CI)

Coming from web tooling, the mental model: `export_presets.cfg` is Godot's
equivalent of a build config (think `vite.config` for the export step). The
editor's *Project → Export* dialog is a GUI over this file. CI never opens the
GUI — it runs `godot --headless --export-release "Web" build/index.html`, which
reads `export_presets.cfg` by the preset **name** (`"Web"`). So:

- Anything you can't see a checkbox for, you can still set by editing the `.cfg`.
- Unknown keys are silently ignored on import and dropped on the next GUI save —
  so you can't invent a knob Godot doesn't have (this bit us for WASM memory).
- The two-pass `--import` then `--export-release` is because a fresh checkout has
  no `.godot/` import cache; the first pass builds it (`|| true` because it can
  exit non-zero while still succeeding), the second exports.

## 2. PWA export: what Godot generates, and the iOS realities

Flipping `progressive_web_app/enabled=true` makes the Web export emit, alongside
`index.html`/`index.wasm`/`index.pck`:

- `index.manifest.json` — the web app manifest (name, icons, `display`,
  `background_color`, `orientation`).
- `index.service.worker.js` — offline caching; also *auto-applies COOP/COEP via
  the SW*. Harmless for us (single-threaded doesn't need isolation — 0002 §6).
- `index.icon.png` / apple-touch-icon wiring from the `icon_144/180/512` fields.

**Icons.** The three PNGs live in `smoke-test/icons/` and are referenced by
`res://` path in the preset. They're generated by `gen_icons.py` — a pure-stdlib
(zlib-only) PNG writer, so no Pillow/ImageMagick needed in the container. For the
real game these become proper art; for a throwaway, a hand-drawn gem is fine.

**iOS gotchas that shaped the config (0002 §6):**

- `display=1` (standalone) is what gives the chrome-less "feels like an app"
  WebClip. Good for us.
- **No Fullscreen API and no orientation lock on iOS.** So `orientation=1`
  (portrait) in the manifest is a *hint* iOS may ignore — we do **not** depend on
  it. The harness instead survives rotation (stretch `canvas_items` + `aspect
  keep`, and it counts resizes to prove it).
- **Installed-PWA storage diverges from Safari's** (one-time copy, then
  independent IndexedDB). Doesn't affect this harness (no saves), but it's why
  0009 exists and why the checklist tests the tab **and** the installed PWA
  separately.

## 3. The WASM-memory lever (the fiddly one — 0002 §4)

The single most important iOS risk in 0002 is memory: the old **2 GB** WASM max
that iOS Safari refused to initialise (godot #70621). Two things a web dev should
internalise:

1. **Godot 4.3's stock templates already fixed the 2 GB problem.** Memory now
   grows dynamically; there is no fixed 2 GB `WebAssembly.Memory({maximum})`
   request to reject. So on a stock 4.3 export there is often *nothing to lower*.
2. **There is no export-preset knob to force a lower max in 4.3.** The "proper"
   way to hard-cap it is rebuilding the export templates with a smaller
   emscripten `MAXIMUM_MEMORY` — far too heavy for a smoke test on a stock CI
   Godot.

So the harness takes the lever that needs no template rebuild: a `<script>` in
`html/head_include` (which Godot injects into `<head>`, running **before** the
engine boots) monkey-patches `WebAssembly.Memory` to **clamp the requested
`maximum` down to 512 MB**. This is the browser-side equivalent of #70621's
reporter lowering `WASM_MEM_MAX`, and it directly exercises the §4 question:
*does iOS Safari accept a modest WASM max at init?*

- The clamp only ever **lowers** the maximum and never below the requested
  `initial`, so it can't cause an init failure by itself.
- **512 MB, not 256**, on purpose: 0002 §4 notes a *hosted* Godot build can sit
  near a **~400 MB** baseline heap, so a 256 MB cap risks a **false OOM** that
  would confound the test. 512 MB clears that baseline while still being a
  quarter of the old 2 GB.
- Readable source + full reasoning: `smoke-test/web/head_include.html`. The
  minified copy in the `.cfg` is what CI actually reads — keep them in sync.

If the shim ever fails to intercept, it's harmless (it only lowers a maximum) and
the on-screen "cap" figure just reads unknown.

## 4. Reading memory live from GDScript — `JavaScriptBridge`

`JavaScriptBridge` is Godot's `eval()` into the page's JS (web platform only —
guard with `OS.has_feature("web")`). It's how a Godot game reaches anything
browser-only. The harness uses it for three readouts:

```gdscript
# WASM linear heap size — WORKS ON SAFARI (buffer.byteLength is standard).
var wasm := float(JavaScriptBridge.eval(
    "(window.__godotWasmMemory && window.__godotWasmMemory.buffer)"
    + " ? window.__godotWasmMemory.buffer.byteLength : -1", true))

# JS heap — Chrome-ONLY; returns -1 on Safari, shown as "n/a (Safari)".
var heap := float(JavaScriptBridge.eval(
    "(window.performance && performance.memory)"
    + " ? performance.memory.usedJSHeapSize : -1", true))

# WebGL context-loss flag, set by a listener installed in head_include.
var lost := bool(JavaScriptBridge.eval("window.__ctxLost === true", true))
```

Key lesson: **`performance.memory` is Chrome-only.** On the *actual target*
(Safari) it's undefined. The Safari-safe memory signal is the WASM buffer's
`byteLength` — which is why the shim stashes the engine's `WebAssembly.Memory` on
`window.__godotWasmMemory` for GDScript to read. Godot's own
`Performance.get_monitor(Performance.MEMORY_STATIC)` is the always-available
cross-check (engine-side static allocation).

Also note the throttle: `eval` isn't free, so the harness polls every 0.25 s, not
every frame.

## 5. Audio: Sample playback + the gesture unlock (0002 §3)

- **Sample vs Stream** is set by `audio/general/default_playback_type.web` (2 =
  Sample) and/or per-node `AudioStreamPlayer.playback_type`. Sample = low latency
  without threads (the reason single-threaded sounds fine), at the cost of no bus
  effects/reverb/doppler/procedural.
- The blip is generated in code as a 16-bit PCM **`AudioStreamWAV`** — a
  *sample*, which is exactly what Sample playback wants. An
  `AudioStreamGenerator` (procedural) would **not** work under web Sample
  playback — good to know before designing 0008's audio.
- **Autoplay unlock:** browsers block audio until a user gesture. The harness
  plays the blip on tap, so the first tap doubles as the unlock — same trick the
  real game's "tap to start" screen will use.

## 6. GitHub Pages / CI wiring — one site, two builds at sub-paths

The constraint that shaped the deploy: **one Pages site per repo, and the
`github-pages` environment only deploys from `main`.** The 0004 prototype already
owns that site at `/`. To add the smoke test without clobbering it, the workflow
(`.github/workflows/deploy-prototype.yml`) now:

1. Exports **both** `prototype/` and `smoke-test/` (two `--export-release`
   steps).
2. Assembles one `site/` dir: `cp prototype/build → site/` and `smoke-test/build
   → site/smoke/`.
3. Uploads `site/` as the single Pages artifact.

Result: prototype at `…/gem-mining-game/`, smoke test at
`…/gem-mining-game/smoke/`. Both are throwaways sharing a site; when each ticket
resolves, delete its folder and drop it from the workflow.

**Branch/deploy split (unchanged from 0004):** the workflow builds on listed
branches for *verification* but the `deploy` job is gated `if: github.ref ==
'refs/heads/main'`. So a feature branch's green build proves the export config is
valid; the live URL only updates once the change is on `main`. That's why getting
a testable URL to Fiachra means merging to `main` (an intentional, reviewed step,
not a silent one — the single live site is shared).

## 7. What a stock CI Godot can't tell us

CI proves the export *builds* and (via a desktop/Chromium open) that it loads,
renders, takes input, and plays sound. It **cannot** reproduce iOS Safari's
memory ceiling or safe-area behaviour — 0002 §8 is explicit that even the iOS
*Simulator* doesn't, only a physical iPhone does. That's the HITL boundary: the
build de-risks everything up to the device, and the device result is Fiachra's to
report. This session did not have a local Godot/iPhone, so the pre-merge signal
is "CI export is green"; the on-device pass/fail lands on the ticket afterward.

## 8. Gotcha that bit us — "export is green" ≠ "the script runs"

The first deploy loaded on the iPhone then showed a blank **grey** screen (that
grey is Godot's *default clear colour* — the engine is up but the scene drew
nothing). Cause: the diagnostic script called `OS.get_current_rendering_method()`
/ `OS.get_current_rendering_driver_name()`. **Those methods don't exist in Godot
4.3** — not on `OS`, and not on `RenderingServer` either (they're a later-4.x
addition). In GDScript a call to a non-existent native method is a **parse
error**, and a parse error makes the *entire script fail to load* — so the scene
node ran no `_ready`/`_process`/`_draw` at all. Silent grey.

Two lessons for a web/TS dev in Godot:

1. **`godot --headless --export-release` compiles-and-packs the script but never
   *runs* it**, so a parse/nonexistent-method error sails straight through a
   green export and only bites when the scene actually loads (on device). Don't
   read "export succeeded" as "the code works."
2. **The fix is a CI gate that RUNS the scene.** The workflow now has a
   *"Headless smoke-run (gate on script errors)"* step: `godot --headless
   --quit-after 120`, then `grep` the log for `SCRIPT ERROR|Parse Error|Failed to
   load script|…` and `exit 1` if found. A green build now means the scene
   *loaded and ran clean*, not merely that it exported. This is the cheap
   stand-in for "no local Godot to press play in."

What 4.3 *does* have for the same info: the configured method from
`ProjectSettings.get_setting("rendering/renderer/rendering_method")`, and the
live WebGL details from `RenderingServer.get_video_adapter_name()` /
`get_video_adapter_api_version()` (the api-version string reads "WebGL 2.0 …" on
device — a nicer WebGL2-up signal than a driver enum anyway).
